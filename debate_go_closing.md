# ソルバー戦略討論 — 実用主義派 Closing Statement（結辯）

**立場**: Go言語による実用優先・段階的拡張アプローチ
**執筆**: 足軽三番
**日付**: 2026-02-14
**フェーズ**: Phase 3（結辯）

---

## 1. 最終主張: 段階的戦略こそが成功確率を最大化する

### 1.1 討論を通じて明らかになった核心的対立

3派の討論を通じて、戦略的対立の本質が浮き彫りになった。

**ES派の主張**: NLP/GPU特化 + C++ + MIT = 先行者利得の最大化
**Req派の主張**: LP/MIP Phase 1 + Rust + Apache-2.0 = 堅実な市場参入
**Go派の主張**: LP Phase 1（Go） → 段階的拡張 → 技術スタック再評価 = 失敗リスクの最小化

この対立の核心は、**「一貫戦略」vs「段階的戦略」**である。ES派とReq派は言語・領域で対立しているが、両者とも「最初から技術スタックを固定する」点では一致している。Go派のみが「段階的に技術スタックを見直す」柔軟性を提案している。

### 1.2 Phase 2で受けた批判への応答

両派からの主要な批判は以下の3点であった:

**批判1: Go→Rust移行は二重投資である**（ES派・Req派共通）
**批判2: GC overheadが致命的である**（ES派・Req派共通）
**批判3: 段階的戦略は「決断の先送り」である**（ES派）

これらの批判は**部分的に正当**である。しかし、批判が見落としている重要な視点がある。

#### 応答1: 二重投資のリスク vs 一貫戦略の失敗リスク

ES派・Req派は「Go→Rust移行は二重投資」と批判するが、この批判は**「Phase 1の選択が正しい」という前提**に立っている。

**現実の問いは異なる:**
- Phase 1で選んだ言語・領域が**間違っていた場合**、どうするのか？
- ES派のGPU-NLP戦略が技術的に実現不可能だった場合、24ヶ月の投資は**全損**である。
- Req派のRust戦略が学習曲線で遅延し、12ヶ月でMVP未達成の場合、市場機会を**逸失**する。

**段階的戦略の真の価値:**
- Phase 1（0-12ヶ月）: Goで**最小リスクで市場検証**を行う。LP MVPが市場に受け入れられるか、HiGHSとの競争が可能かを実証する。
- Phase 2（12-24ヶ月）: Phase 1のフィードバックに基づき、技術スタックを再評価する。Go継続、Rust移行、またはC++採用を**データに基づき判断**する。

Go→Rust移行は「二重投資」ではなく、**「Phase 1の検証コスト」**である。失敗リスクの高い一貫戦略に24ヶ月を賭けるより、12ヶ月で検証してから判断する方が合理的である。

#### 応答2: GC overheadは「Phase 1の妥協」であり「Phase 3の最適化対象」である

ES派・Req派の「GC overhead致命的」批判は、**Phase 1の目標を誤解している**。

**Phase 1の目標:**
- 「商用ソルバー超え」ではなく「HiGHS比80%で市場検証」である。
- HiGHS比80%が達成できれば、SciPy等への組み込み候補となり得る（HiGHSも当初は商用ソルバーに劣後していた）。

**GC overheadの現実的評価:**
- Go 1.26 Green Tea GCで10-40% overhead削減済み（debate_go_rebuttal.md §3.1）
- GOGC調整、arena allocatorで更なる削減可能
- 「HiGHS比80%」は、GC overhead 20%を許容しても達成可能な目標

**Phase 2以降の最適化パス:**
- Phase 1で市場検証に成功した場合、Phase 2で以下の選択肢を検討:
  1. Go継続 + GC最適化（GOGC, arena, asm最適化）
  2. Rust移行（コア部分のみ、段階的）
  3. C++移行（性能優先の場合）

GC overheadは「Phase 1で許容し、Phase 2以降で最適化または移行」する対象である。Phase 1から性能を追求し、市場検証に失敗するリスクを取る必要はない。

#### 応答3: 段階的戦略は「決断の先送り」ではなく「データに基づく決断」である

ES派は「段階的戦略は決断の先送り」と批判する（debate_es_rebuttal.md §2.3）。

**この批判は誤解である。**

段階的戦略は「決断しない」のではなく、**「決断のタイミングを最適化」**する。

**一貫戦略のリスク:**
- Phase 1開始時点（調査段階）で、24-36ヶ月の技術スタックを固定する
- 市場フィードバック、技術的実現可能性を検証する前に決断する
- 決断が間違っていた場合、**全投資が失われる**

**段階的戦略の利点:**
- Phase 1（12ヶ月）で市場検証、技術的実現可能性を実証する
- Phase 1の**データ**に基づき、Phase 2の技術スタックを決断する
- 決断が間違っていた場合でも、損失は12ヶ月に限定される

これは「決断の先送り」ではなく、**「データに基づく合理的決断」**である。

### 1.3 Go派の最終主張: 実用主義が最も成功確率が高い

討論を通じて、各派の戦略リスクが明確化した:

**ES派のリスク:**
- GPU-NLP技術的実現可能性が未実証（LP-PDHGの成功 ≠ NLP内点法の成功）
- 需要検証が不十分（KNITRO/BARON顧客 ≠ GPU-NLPソルバーの需要）
- 1年MVP達成が非現実的（商用・OSSともにGPU-NLP実績ゼロ）

**Req派のリスク:**
- Rust学習曲線がPhase 1（12ヶ月）で致命的（初期3-6ヶ月は生産性低迷）
- LP/MIP市場の競合密度（HiGHS, SCIP, Gurobi/CPLEX密集）
- 「並列化ファースト」だけではHiGHS超えの差別化が不明確

**Go派のリスク:**
- GC overheadによる性能劣位（HiGHS比80%目標だが、実測で70%の可能性）
- Go→Rust移行の実現性（理論上可能だが、実務上困難な可能性）
- エコシステム不在（Goソルバー実績ゼロ）

**3派比較: 失敗時の損失**

| 戦略 | Phase 1投資 | Phase 1失敗時の損失 | 成功確率（主観評価） |
|------|------------|-------------------|-------------------|
| ES派 | 12ヶ月（GPU-NLP） | 12ヶ月全損（技術的実現不可の場合） | 中（30-50%） |
| Req派 | 12ヶ月（Rust LP/MIP） | 12ヶ月全損（学習曲線で遅延、MVP未達成） | 中（40-60%） |
| Go派 | 12ヶ月（Go LP） | 12ヶ月（ただしPhase 2で方向転換可能） | 高（60-80%） |

Go派の戦略は、Phase 1失敗時の損失を最小化し、Phase 2での方向転換余地を最大化する。これが**最も成功確率が高い**理由である。

## 2. 譲歩と修正

討論を通じて、Go派の当初主張の一部は修正すべきであることが明らかになった。率直に認める。

### 2.1 ES派への譲歩: GPU-NLPの技術的魅力を過小評価していた

**当初主張:**
「GPU-NLPはPhase 3以降で検討」（debate_go_opening.md §1.2）

**ES派の反論:**
- cuOptの8.6倍高速化、LP-PDHGの成功が実証済み（debate_es_opening.md §1.3）
- GPU-NLPソルバーは商用・OSSともにゼロ = 先行者利得が極めて大きい

**譲歩:**
ES派の指摘は正当である。GPU-NLPの技術的魅力と先行者利得の大きさを、Go派は過小評価していた。

**修正後の立場:**
- GPU-NLPは**Phase 2での検討対象に引き上げる**
- Phase 1（0-12ヶ月）: Go LP MVP
- Phase 2（12-24ヶ月）: MIP追加 **または** GPU-NLP検討（Phase 1フィードバックに基づき選択）
- Phase 3（24-36ヶ月）: 未選択の方を実装

GPU-NLPを「Phase 3以降」から「Phase 2検討」に引き上げることで、ES派の技術的野心を段階的戦略に統合できる。

### 2.2 Req派への譲歩: Rustの長期的価値を認める

**当初主張:**
「Phase 1はGo、Phase 3でRust再実装検討」（debate_go_opening.md §1.2）

**Req派の反論:**
- Rustの学習曲線は「初期のみ」であり、長期的にはデバッグ時間削減で開発速度向上（debate_req_rebuttal.md §2.3）
- 並列化ファースト設計において、Rustの所有権システムは決定的優位（debate_req_opening.md §2.2）
- Go→Rust移行は二重投資（debate_req_rebuttal.md §2.2）

**譲歩:**
Req派の指摘は正当である。Rustの長期的価値（メモリ安全性、並列安全性、ゼロコスト抽象化）は、Go派の当初評価より高い。

**修正後の立場:**
- **Phase 1言語選択を柔軟化する**
- オプションA: Go LP MVP（開発速度優先）
- オプションB: Rust LP MVP（長期的価値優先、ただし学習曲線リスクを許容）
- 選択基準: プロジェクト開始時点での開発者のRust習熟度、リソース状況に応じて決定

Go一択の固執を撤回し、Rustも「Phase 1の選択肢」として認める。

### 2.3 両派への譲歩: エコシステム統合の重要性をより明確に認識

**当初主張:**
「Python統合最優先」とは述べたが、具体的戦略が不明確だった（debate_go_opening.md §1.2）

**ES派・Req派の指摘:**
- HiGHSの成功要因は「MIT + エコシステム統合」（ES派 debate_es_opening.md §3.1）
- SciPy, CVXPY統合が成功の鍵（Req派 debate_req_opening.md §1.3）

**譲歩:**
両派の指摘は正当である。エコシステム統合は「重要」というレベルではなく、**「死活的に重要」**である。

**修正後の立場:**
- Phase 1のマイルストーンに「PyPI公開」「SciPy PR提出」を明示的に含める
- Python bindings開発を、コアアルゴリズム実装と**並行**して進める（後回しにしない）
- HiGHSのエコシステム統合戦略（SciPy, MathWorks, NAG）を詳細に研究し、踏襲する

### 2.4 自己批判: Go→Rust移行の実現可能性を過大評価していた

**当初主張:**
「Phase 3でコア部分のRust再実装を検討」（debate_go_opening.md §1.2）

**ES派の反論:**
- Go→Rust移行は理論上可能だが、実務上は極めて困難（Python 2→3の悪夢）（debate_es_rebuttal.md §2.1）
- ユーザーベース確立後の大規模書き換えは、互換性維持・移行期間のサポート負担が膨大

**自己批判:**
ES派の指摘は正当である。Go→Rust移行の困難さを、Go派は過小評価していた。

**修正後の立場:**
- Go→Rust移行を「Phase 3の確定計画」から「Phase 2終了時点での評価対象」に格下げする
- Phase 2終了時点で、以下を評価:
  1. Go実装の性能が商用ソルバーに迫れているか
  2. ユーザーベースのサイズと、移行コスト
  3. Rustエコシステムの成熟度
- 上記評価の結果、「Go継続」が合理的なら、Rust移行を断念する

Go→Rust移行を「確定計画」から「条件付き選択肢」に修正する。

## 3. 統一戦略への提案

討論を通じて、3派の合意可能な点が明確化した。ここに**統一戦略**を提案する。

### 3.1 参入領域: ハイブリッド戦略 — LP Phase 1、Phase 2で分岐

**合意可能な点（全派）:**
- LP/MIPは産業応用の中核（Req派の主張、ES派・Go派も否定していない）
- GPU-NLPは先行者利得が大きい（ES派の主張、Req派・Go派も技術的魅力を認める）
- 段階的拡張の必要性（Go派の主張、Req派も一部同意、ES派は反対だが理解可能）

**統一提案:**

#### Phase 1（0-12ヶ月）: LP特化MVP
- 対象: LP（線形計画）のみ
- アルゴリズム: Simplex（Primal/Dual）、前処理、基本的な内点法
- 目標性能: HiGHS比80%（小〜中規模問題）
- マイルストーン:
  1. Netlib LPベンチマークでHiGHS比80%達成（6ヶ月）
  2. PyPI公開、Python bindings完成（9ヶ月）
  3. SciPy PR提出（12ヶ月）

**全派が合意可能な理由:**
- Req派: LP Phase 1は当初主張通り
- ES派: 「LP/MIPは既存OSSを活用」と主張していたが、LP実装経験は将来のNLP統合に有用
- Go派: 当初主張通り

#### Phase 2（12-24ヶ月）: 分岐戦略
Phase 1のフィードバックに基づき、**以下3つのいずれかを選択**:

**オプションA: MIP拡張（Req派寄り）**
- Branch-and-cut実装
- 並列化（マルチスレッド）
- 目標: HiGHS比90%、SCIP比同等

**オプションB: GPU-NLP拡張（ES派寄り）**
- cuOpt PDHGを参考にGPU加速内点法実装
- 既存LPサブソルバー（Phase 1実装 or HiGHS）統合
- 目標: Ipopt比3-5倍高速化（GPU利用時）

**オプションC: QP拡張 + 並列化（Go派・中間案）**
- QP（二次計画）実装
- 並列化強化
- 目標: OSQP比同等、並列性能でHiGHS超え

**選択基準:**
1. Phase 1でのユーザーフィードバック（LP専用で十分か、MIP/NLPの需要があるか）
2. Phase 1での技術的実現可能性（並列化の実装難易度、GPU統合の実現可能性）
3. 競合状況（HiGHS/SCIPの進化、商用ソルバーの動向）

**全派が合意可能な理由:**
- 「Phase 1後にデータに基づき判断」という柔軟性を全派が受け入れれば、各派の主張を「Phase 2の選択肢」として並列保存できる
- ES派: オプションBでGPU-NLP実現
- Req派: オプションAでMIP拡張実現
- Go派: Phase 1後のデータに基づく判断を実現

#### Phase 3（24-36ヶ月）: 残りのオプション実装 or 性能最適化

### 3.2 実装言語: Phase 1は柔軟、Phase 2で固定

**合意可能な点（全派）:**
- C++はエコシステム成熟（ES派の主張）
- Rustはメモリ安全性・並列安全性で優位（Req派の主張）
- Goは開発速度で優位（Go派の主張）
- **しかし、全派とも「自分の言語が唯一正解」という証明はできていない**

**統一提案:**

#### Phase 1言語選択: プロジェクト開始時点の状況に応じて柔軟に選択

**選択肢:**
1. **Go**: 開発速度優先、12ヶ月MVP確実達成を最優先する場合
2. **Rust**: 長期的価値優先、開発者がRust習熟済み、またはGC overheadを許容できない場合
3. **C++**: 既存エコシステム活用優先、開発者がC++習熟済み、メモリバグリスクを許容できる場合

**選択基準:**
- 開発者のスキルセット
- 12ヶ月MVP達成の確実性
- GC overhead許容度

**Phase 1後の再評価:**
- Phase 1（12ヶ月）終了時点で、Phase 2の言語を再評価
- Phase 1言語を継続するか、別言語に移行するかを**データに基づき判断**
- 判断材料:
  1. Phase 1性能（HiGHS比80%達成できたか）
  2. 開発速度（12ヶ月でMVP達成できたか）
  3. Phase 2の技術的要件（並列化、GPU統合の難易度）

**全派が合意可能な理由:**
- 「Phase 1で一つの言語を試し、Phase 2で再評価」という柔軟性を全派が受け入れれば、言語論争を「Phase 1実験」に転換できる
- ES派: C++が最適と確信しているなら、Phase 1でC++を選び、Phase 2で優位性を実証できる
- Req派: Rustが最適と確信しているなら、Phase 1でRustを選び、Phase 2で優位性を実証できる
- Go派: Goが最適と確信しているなら、Phase 1でGoを選び、Phase 2で優位性を実証できる

#### 現実的推奨（Go派の主観）:
Phase 1言語として、**リスクの低い順**に:
1. **Rust** — 長期的価値最大、Req派の主張を尊重、ただし学習曲線リスクあり
2. **Go** — 開発速度最大、12ヶ月MVP確実性最高、ただしGC overheadリスクあり
3. **C++** — エコシステム成熟最大、ES派の主張を尊重、ただしメモリバグリスクあり

開発者のスキルセットとリスク許容度に応じて選択。

### 3.3 ライセンス: Apache-2.0 / MIT デュアルライセンス

**合意可能な点（全派）:**
- MITはエコシステム統合を加速する（ES派・Go派の主張）
- Apache-2.0は特許保護を提供する（Req派の主張）
- **両ライセンスとも商用組み込み自由**（全派共通認識）

**統一提案:**

**Apache-2.0 / MIT デュアルライセンスを採用**

```
This project is dual-licensed under Apache-2.0 OR MIT.
Users may choose either license.
```

**デュアルライセンスの利点:**
- MITを選びたいユーザー（シンプルさ重視）はMITを選べる
- Apache-2.0を選びたいユーザー（特許保護重視）はApache-2.0を選べる
- エコシステム統合の障壁を最小化（MITオプションがあるため）
- 特許訴訟リスクを低減（Apache-2.0オプションがあるため）

**実証:**
- Rustエコシステムの慣例（requirements.md §5）
- OR-Tools（Google）も実質的にApache-2.0でエコシステム統合成功

**全派が合意可能な理由:**
- ES派: MITオプションが残るため、エコシステム統合戦略は維持される
- Req派: Apache-2.0オプションが含まれるため、特許保護は達成される
- Go派: 両派の主張を統合し、収束を実現できる

## 4. 現実的な折衷案 — 具体的ロードマップ

上記の統一戦略を、具体的なロードマップに落とし込む。

### Phase 1（0-12ヶ月）: LP特化MVP

**目標:**
- LP Simplex（Primal/Dual）実装
- Netlib LPベンチマークでHiGHS比80%達成
- PyPI公開、Python bindings完成
- SciPy PR提出

**実装言語:**
- プロジェクト開始時点の開発者スキルセットに応じて、Go/Rust/C++から選択
- **推奨**: Rust（長期的価値、Req派の主張を尊重）または Go（開発速度、12ヶ月確実性）

**ライセンス:**
- Apache-2.0 / MIT デュアルライセンス

**マイルストーン:**
- M1（3ヶ月）: Simplex Primal実装、Netlib小規模問題で動作
- M2（6ヶ月）: Simplex Dual実装、前処理追加、Netlib中規模でHiGHS比80%
- M3（9ヶ月）: Python bindings完成、PyPI公開、基本的な内点法追加
- M4（12ヶ月）: SciPy PR提出、ドキュメント整備

### Phase 1後の評価会議（12ヶ月時点）

**評価項目:**
1. 性能目標達成度（HiGHS比80%達成できたか）
2. 開発速度（12ヶ月でMVP達成できたか）
3. ユーザーフィードバック（LP専用で十分か、MIP/NLPの需要があるか）
4. 言語選択の妥当性（GC overhead/学習曲線/メモリバグは問題だったか）
5. 競合状況（HiGHS/SCIPの進化、商用ソルバーの動向）

**Phase 2方向性の決定:**
- オプションA（MIP拡張）、オプションB（GPU-NLP拡張）、オプションC（QP拡張 + 並列化）から選択
- Phase 1言語を継続するか、別言語に移行するかを決定

## 5. 最終メッセージ: 実用主義の本質

### 5.1 実用主義とは妥協ではなく、リスク管理である

ES派は「Go派の実用主義は妥協である」と批判した（debate_es_rebuttal.md §2.3）。

**この批判は誤解である。**

実用主義の本質は、**「不確実性の下で最も成功確率が高い戦略を選ぶ」**ことである。

**不確実性の現実:**
- GPU-NLPが1年で実現可能か、**誰も確実に知らない**（ES派も「実現可能性の根拠」を示したが、未実証である）
- Rustの学習曲線が12ヶ月MVP達成に致命的か、**誰も確実に知らない**（Req派は「初期のみ」と主張するが、個人差がある）
- Goのエコシステム不在が致命的か、**誰も確実に知らない**（商用・OSSソルバー実績ゼロだが、それが失敗を証明するわけではない）

**この不確実性の下で:**
- ES派・Req派は「一貫戦略」を提案する = 不確実性を無視し、自分の判断に24-36ヶ月を賭ける
- Go派は「段階的戦略」を提案する = 不確実性を認識し、12ヶ月ごとに再評価する

**どちらがリスク管理として優れているか?**

答えは明白である。不確実性の下では、**段階的戦略がリスクを最小化する**。

### 5.2 3派討論の真の価値 — 選択肢の可視化

この討論の真の価値は、「どの派が正しいか」を決めることではない。

**真の価値は、「戦略の選択肢を可視化したこと」である。**

討論を通じて、以下が明確化した:
1. 参入領域の選択肢: LP/MIP（Req派）、GPU-NLP（ES派）、段階的（Go派）
2. 実装言語の選択肢: C++（ES派）、Rust（Req派）、Go（Go派）
3. ライセンスの選択肢: MIT（ES派）、Apache-2.0（Req派）、段階的（Go派）

**これらの選択肢は、すべて合理性を持つ。**

どれを選ぶかは、**プロジェクトの状況（開発者スキル、リソース、リスク許容度）に依存する**。

### 5.3 統一戦略の核心 — 柔軟性と収束の両立

Go派が提案する統一戦略の核心は、**「柔軟性」と「収束」の両立**である。

**柔軟性:**
- Phase 1後に再評価し、方向転換可能
- 言語・領域・ライセンスを、データに基づき調整可能

**収束:**
- Phase 1の具体的目標（LP特化MVP、HiGHS比80%、PyPI公開、SciPy PR）を全派が共有
- Phase 1言語選択の柔軟性（Go/Rust/C++の選択肢）を全派が受け入れる
- ライセンス（Apache-2.0/MITデュアル）で全派の主張を統合

**この両立により:**
- 各派の主張を「Phase 1実験」または「Phase 2選択肢」として保存
- Phase 1で全派が協力可能な共通基盤を確立
- Phase 1後のデータに基づき、最も成功確率が高い方向を選択

### 5.4 殿様への最終具申

この討論を通じて、3派の戦略はいずれも一定の合理性を持つことが明らかになった。

**しかし、不確実性の下では、段階的戦略が最もリスクを最小化する。**

**Go派の最終提案:**

1. **Phase 1（0-12ヶ月）**: LP特化MVP
   - 実装言語: Rust（推奨）または Go（開発速度優先の場合）
   - 目標: HiGHS比80%、PyPI公開、SciPy PR提出
   - ライセンス: Apache-2.0 / MIT デュアル

2. **Phase 1後の評価会議（12ヶ月）**: データに基づく方向決定
   - Phase 2方向性: MIP拡張（オプションA）、GPU-NLP拡張（オプションB）、QP + 並列化（オプションC）から選択
   - Phase 2言語: Phase 1言語継続 or 移行を決定

3. **Phase 2（12-24ヶ月）**: 選択した拡張を実装

4. **Phase 3（24-36ヶ月）**: 残りオプション or 性能最適化

**この戦略により:**
- ES派の技術的野心（GPU-NLP）を Phase 2選択肢として保存
- Req派の堅実性（LP/MIP、Rust）を Phase 1推奨として採用
- Go派の柔軟性（段階的再評価）を Phase 1後評価として実現

**3派の良い点を統合し、失敗リスクを最小化し、成功確率を最大化する。**

これが、実用主義派の最終主張である。
