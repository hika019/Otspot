## 1. 検収サマリ

### 1.1 討論の質の評価

**論拠の充実度: 高**

3派とも市場データ（HiGHS採用実績、BARON/KNITRO顧客数、SciPyダウンロード数、cuOpt性能実績）を具体的に引用し、抽象論に逃げなかった。ES派のGPU-NLP市場分析、Req派のHiGHS弱点分析（シングルスレッド中心）、Go派のリスク定量評価（成功確率の主観推定）はいずれも具体性がある。

**相互批判の深さ: 非常に高い**

各派がPhase 2（反駁）で相手の核心的弱点を正確に突いた。特に以下の指摘が鋭い:
- Req派・Go派 → ES派: 「LP-PDHGの成功をNLP内点法に外挿するのは技術的飛躍」（ES派自身がClosingで承認）
- ES派・Go派 → Req派: 「Rust学習曲線3-6ヶ月はPhase 1の12ヶ月MVPを脅かす」（Req派がClosingで修正案提示）
- ES派・Req派 → Go派: 「Go→Rust移行は二重投資」（Go派がClosingで移行計画を条件付き選択肢に格下げ）

**収束の度合い: 高い**

Closingにおいて3派とも大幅な譲歩を行い、以下の3点で実質合意に達した:
1. Phase 1はLP/MIPから始める（ES派がGPU-NLP Phase 1を撤回）
2. 段階的拡張が必要（ES派が一括戦略を撤回）
3. ライセンスはデュアルまたは段階移行（全派が収束）

### 1.2 各派の強み・弱みの総括

| 派 | 強み | 弱み |
|----|------|------|
| **ES派** | NLP/GPU市場空白の発見、MITの戦略的価値の論証、C++エコシステム分析 | Phase 1戦略が非現実的（GPU-NLP 1年MVP）、需要検証不足、C++一択の視野狭窄 |
| **Req派** | LP/MIP市場分析の堅実さ、並列化ファースト設計の着眼点、Rust安全性の技術的論証 | 学習曲線コストの過小評価、差別化戦略の具体性不足（Closingで修正） |
| **Go派** | リスク管理フレームワークの明確さ、段階的戦略の合理性、柔軟性と収束の両立 | GC overheadへの楽観（HiGHS比80%の根拠が弱い）、Go→Rust移行の実現性過大評価（Closingで修正） |

**総評**: ES派は**ビジョンの質**が最高（GPU-NLP市場空白の発見は本物の洞察）。Req派は**技術的堅実さ**が最高（並列安全性・ライセンス分析）。Go派は**戦略的思考**が最高（リスク管理・段階的判断）。3派の討論は対立ではなく相互補完であり、単独のどの派よりも強固な戦略が導出可能である。

## 2. 参入領域の決定

### 最終結論: LP Phase 1 → MIP Phase 2 → NLP/GPU Phase 3

**Phase 1でLP（線形計画）に集中する。**

### 根拠

**Req派の主張を主軸に採用（80%）、Go派の段階思想を統合（15%）、ES派のビジョンをPhase 3に保存（5%）**

1. **市場規模**: LP/MIPは産業応用の中核（生産計画・輸送最適化・スケジューリング）。SciPy月間数千万DLという巨大エコシステムへの統合が最大の成長レバーとなる（Req派Opening §1.1）。ES派のNLP市場（BARON 1,000+顧客）は実在するが、LP/MIP市場の1桁下である。

2. **技術的実現可能性**: LP Simplexは理論・実装とも成熟しており、12ヶ月MVPの実現確度が高い。GPU-NLP（ES派当初案）はPDHG理論がNLPで未確立であり、Phase 1としてはリスク過大（ES派自身がClosingで撤回）。

3. **実証済みモデル**: HiGHSがLP/MIP特化→SciPy/MathWorks/NAG採用という成功パターンを既に実証している。この成功経路を踏襲することが最も確実。

4. **段階的拡張の合理性**: 3派すべてがClosingで段階的拡張に合意。Phase 1で市場検証→Phase 2でフィードバックに基づく拡張という流れは、Go派の「データに基づく判断」思想と合致する。

### フェーズ計画

**Phase 1（0-12ヶ月）: LP Simplex MVP**
- 実装範囲: Primal/Dual Simplex、前処理、基本的な内点法
- 性能目標: Netlib LPベンチマークでHiGHS比80%以上（小〜中規模問題）
- エコシステム: Python bindings + PyPI公開（9ヶ月）、SciPy PR提出（12ヶ月）
- マイルストーン: M1(3ヶ月)動作確認 → M2(6ヶ月)性能達成 → M3(9ヶ月)PyPI → M4(12ヶ月)SciPy PR

**Phase 2（12-24ヶ月）: MIP Branch-and-cut + CPU並列化**
- 実装範囲: Branch-and-cut、並列ツリー探索（HiGHSの最大弱点を突く）
- 性能目標: MIPLIB 2017でHiGHS比90%以上
- 差別化: 並列化ファースト設計（Req派の着眼点を採用）
- 評価会議: Phase 2終了時にPhase 3方向を決定

**Phase 3（24-36ヶ月）: NLP/GPU拡張**
- Phase 1-2のユーザーフィードバックに基づき、以下から選択:
  - A) GPU加速LP/MIP（PDHG実装、ES派のビジョン）
  - B) NLP拡張（SQP法、内点法）
  - C) QP拡張 + 性能最適化
- **前提条件**: Phase 3開始前に化学・エネルギー企業ヒアリングを実施し、需要を検証する（ES派Closingで需要検証不足を認めた点を踏まえる）

## 3. 実装言語の決定

### 最終結論: Rust

### 根拠

**Req派の主張を主軸に採用（60%）、ES派のエコシステム論を参考に（20%）、Go派の段階思想で緩和策を設計（20%）**

3派の言語論争は討論だけでは収束しなかった（ES派Closing §3.2が正しく指摘）。しかし、検収裁定者として判断を下す。

1. **並列安全性が決定的差別化**: Phase 2の並列ツリー探索はプロジェクトの最大差別化ポイントである。HiGHSがシングルスレッド中心に留まる理由はC++での並列化デバッグの困難さにある（Req派Opening §2.2）。Rustの所有権システムはデータ競合をコンパイル時に防止し、この問題を構造的に解消する。

2. **メモリ安全性の長期的価値**: ソルバーは長時間実行・大規模データ処理が常態であり、メモリバグの影響が深刻。C++のAddressSanitizer/Valgrindは「事後検出」だが、Rustは「事前防止」。金融・航空宇宙等のセキュリティ意識の高い企業への訴求力にもなる（Req派Closing §2.3）。

3. **学習曲線の緩和策**: Rust学習曲線（3-6ヶ月）は実在するリスクだが、以下で緩和可能:
   - スケジュールに明示的に組み込む（0-3ヶ月を学習+設計期間とする）
   - Rust経験者を初期チームに含める（russcip, nalgebraコントリビューター等）
   - Go派提案のプロトタイプ手法を部分採用: アルゴリズム設計はPython/擬似コードで先行検証し、Rust実装に集中

4. **C++を選ばない理由**: エコシステム成熟度（全主要ソルバーがC++）は事実だが、メモリバグ・並列化困難というコストがPhase 2以降で顕在化する。ES派自身がClosingでC++一択を撤回し「プロトタイプ検証」に後退した。

5. **Goを選ばない理由**: 開発速度の優位性は認めるが、GC overheadがPhase 2以降の性能限界となる。Go派自身がClosingでGo→Rust移行の困難さを認め、Phase 1言語としてRustを「推奨」に引き上げた（Go派Closing §3.2）。Go派がRustを推奨した事実は重い。

### リスク緩和策

| リスク | 緩和策 |
|--------|--------|
| 学習曲線によるPhase 1遅延 | 0-3ヶ月を学習+設計期間として明示的に確保。Rust経験者を初期チームに含める |
| エコシステム未成熟 | nalgebra/ndarrayは成長中。russcip（SCIP Rustバインディング）が活発開発中。Mosekも Rust API提供 |
| FFI複雑性（Python bindings） | PyO3が成熟しており、Rust→Python FFIは実用レベル |
| Phase 1でMVP未達の場合 | 12ヶ月時点の評価会議で、Go移行（性能妥協してでもMVP完成）を検討する余地を残す |

## 4. ライセンスの決定

### 最終結論: Apache-2.0 / MIT デュアルライセンス（Phase 1から）

### 根拠

**Req派の主張を主軸に採用（50%）、ES派・Go派の採用促進論を統合（50%）**

1. **デュアルライセンスが全派の要求を満たす**: MIT（採用促進、ES派・Go派の要求）とApache-2.0（特許保護、Req派の要求）を利用者が選択可能。

2. **Rustエコシステムの慣例に合致**: Rustクレートの標準慣行がApache-2.0/MITデュアルであり、言語選択（Rust）と整合する。

3. **Phase 1からデュアルにする理由**: ES派・Go派は「MIT先行→Phase 2でデュアル化」を提案したが、デュアルライセンスでもMITを選べるため採用障壁は増えない。一方、後からApache-2.0を追加する場合、全コントリビューターの同意が必要になるリスクがある。最初からデュアルにする方が変更コストが低い。

4. **SCIPの先例**: SCIP 10.0（2025年11月）がApache-2.0/LGPLデュアルに移行。業界トレンドとして特許保護の重要性が高まっている。

## 5. 統一ロードマップ

### Phase 1（0-12ヶ月）: LP Simplex MVP

| 期間 | マイルストーン | 内容 |
|------|--------------|------|
| 0-3ヶ月 | M1: 基盤構築 | Rust学習+設計、Primal Simplex実装、Netlib小規模問題で動作確認 |
| 3-6ヶ月 | M2: 性能達成 | Dual Simplex実装、前処理追加、Netlib中規模でHiGHS比80%達成 |
| 6-9ヶ月 | M3: エコシステム | Python bindings（PyO3）完成、PyPI公開、基本的な内点法追加 |
| 9-12ヶ月 | M4: 市場投入 | SciPy PR提出、ドキュメント整備、Mittelmann LP部門で性能実証 |

- **言語**: Rust
- **ライセンス**: Apache-2.0 / MIT デュアル
- **性能目標**: HiGHS比80%以上（小〜中規模LP）
- **成功基準**: PyPI公開完了 + SciPy PR提出 + Netlib/Mittlemannベンチマーク公開

### Phase 1後評価会議（12ヶ月時点）

以下を評価し、Phase 2方向を決定:
1. 性能目標達成度
2. 開発速度（12ヶ月でMVP達成できたか）
3. ユーザーフィードバック（LP専用で十分か、MIP/NLPの需要があるか）
4. Rust選択の妥当性（学習曲線は問題だったか）
5. 競合動向（HiGHS/SCIPの進化）

### Phase 2（12-24ヶ月）: MIP並列化

| 期間 | マイルストーン | 内容 |
|------|--------------|------|
| 12-15ヶ月 | M5: MIP基盤 | Branch-and-bound基本実装 |
| 15-18ヶ月 | M6: Cut強化 | Cutting planes（Gomory, MIR等）追加 |
| 18-21ヶ月 | M7: 並列化 | 並列ツリー探索実装（Rustの並列安全性を活用） |
| 21-24ヶ月 | M8: 最適化 | MIPLIB 2017ベンチマーク、HiGHS比90%目標 |

- **差別化ポイント**: 並列化ファースト設計（HiGHSの最大弱点を突く）
- **ライセンス**: Phase 1継続（Apache-2.0/MITデュアル）
- **成功基準**: MIPLIB 2017でHiGHS比90% + 並列性能でHiGHS超え + CVXPY統合PR

### Phase 3（24-36ヶ月）: NLP/GPU拡張

Phase 2評価に基づき以下から選択:

**オプションA: GPU加速LP/MIP（ES派ビジョン）**
- cuOpt PDHGモデルを参考にGPU加速実装
- Rust + CUDA FFI（cudarc等）
- 目標: 大規模LP/MIPで既存CPU実装の3-5倍高速化

**オプションB: NLP拡張**
- SQP法、内点法の実装
- 既存LP/MIPサブソルバーとの統合
- 目標: Ipopt比同等以上

**オプションC: 性能最適化 + エコシステム拡充**
- LP/MIPの性能を商用ソルバー水準に引き上げ
- Julia/R/C++ bindings追加
- 目標: Gurobi/CPLEX比50%以上

**選択前提条件**: Phase 3開始前に需要検証（化学・エネルギー企業ヒアリング）を実施する。

## 6. 残課題（殿の判断が必要な項目）

### 6.1 言語選択の最終承認

本決定書はRustを推奨するが、以下の場合は殿の判断で変更可能:
- 初期チームにRust経験者が確保できない場合 → Go（開発速度優先）に変更
- 既存C++資産を活用したい場合 → C++に変更

ES派提案の「並行プロトタイプ実装」（3言語で3ヶ月競争）も選択肢として残る。ただし、リソースが3倍必要になるため、チーム規模次第。

### 6.2 Phase 1の性能目標水準

HiGHS比80%を目標としたが、SciPy統合のためにはどの水準が最低限必要か、SciPyコミュニティとの事前調整が必要。HiGHS比50%でも統合可能か、80%が最低ラインかで、Phase 1のスコープが変わる。

### 6.3 チーム編成・リソース

本決定書は戦略の方向性を定めたが、以下は未決定:
- 開発者の人数・スキルセット
- 専任か兼務か
- 外部コントリビューター獲得戦略

### 6.4 プロジェクト名・リポジトリ

OSSプロジェクトとしての名前、GitHub Organization、ブランディング戦略は未定。HiGHSの成功パターン（覚えやすい名前 + 明確なポジショニング）を参考にすべき。

### 6.5 GPU-NLP需要検証の時期

Phase 3の方向決定にはNLP市場の需要検証が必要。Phase 2中に並行してヒアリングを開始するか、Phase 2完了後に着手するかは殿の判断。早期着手を推奨する。

## 付録: 討論プロセスの評価

### 3派討論方式の有効性

本討論は、単独の調査・分析では到達できない深度に達した。具体的な成果:

1. **ES派の過大リスクの発見**: GPU-NLP 1年MVPの非現実性は、Req派・Go派の批判なしには見過ごされた可能性が高い
2. **言語選択の多角的評価**: C++/Rust/Goの長短が、各推進者と批判者の両面から分析され、偏りのない評価が可能になった
3. **段階的戦略への収束**: ES派の野心的ビジョン + Req派の堅実性 + Go派のリスク管理が統合され、単独派よりも強固な戦略が生まれた

### 残された論争点

言語選択は討論では完全に収束しなかった。本決定書ではRustを選択したが、これはClosing時点で3派中2派（Req派: Rust推奨、Go派: Rust推奨に引き上げ）がRustを支持した事実に基づく。ES派のC++エコシステム論も一定の合理性があり、殿の最終判断を仰ぐ。
