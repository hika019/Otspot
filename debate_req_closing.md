**立場**: LP/MIP Phase 1 + Rust + Apache-2.0
**論客**: 足軽二番

## 1. 最終主張: Req派の結論

Req派は、3フェーズにわたる討論を通じて、以下の立場を堅守する。

### 1.1 参入領域: LP/MIP Phase 1は戦略的に正しい

**理由1: 市場規模が最大**

- LP/MIPは産業応用の中核（生産計画、輸送最適化、スケジューリング等）
- ES派のNLP特化は「ブルーオーシャン」ではなく「小さな池」である
- BARON顧客1,000超は、LP/MIP市場（SciPy月間数千万DL）と比べて**絶対数として1桁小さい**

**理由2: HiGHSの成功が一点突破戦略の有効性を実証**

- HiGHSはLP/MIP特化でSciPy/MathWorks/NAGに採用された
- 「既存OSSを活用してNLP特化」(ES派提案) は、実際には**Bonminが既に実現済み**であり、差別化にならない
- HiGHSの成功要因は「MIT + エコシステム統合」だが、**LP/MIPという中核市場をターゲット**にしたことが大前提

**理由3: 商用ソルバーとの性能差が最大チャンス**

- HiGHS（OSS最速）はGurobi/CPLEXより20-30倍遅い → 改善余地が大きい
- この性能差は「商用の堀が深い」ことを示すが、同時に**OSSの伸びしろが大きい**ことも示す
- 並列化ファースト設計により、HiGHSの弱点（シングルスレッド中心）を突ける

**反論への応答:**
- ES派は「LP/MIPはレッドオーシャン」と主張するが、**HiGHSがSciPyに採用された事実**が、OSS参入の余地を証明している
- Go派は「LP Phase 1」で方向性を共有しているが、言語選択（Go）が誤りである

---

### 1.2 実装言語: Rustは長期的投資として最適

**理由1: メモリ安全性・並列安全性がコンパイル時に保証される**

- C++のメモリバグ（use-after-free, buffer overflow等）は、デバッグに数日〜数週間を要する
- ES派は「適切なコーディング規約・テスト・レビューがあれば十分安全」と主張するが、これは**追加的コスト**である
- Rustではコンパイラが保証する → デバッグ時間が劇的に短縮

**理由2: 並列化での決定的優位**

- HiGHSがシングルスレッド中心である理由は、C++での並列化が**デバッグ困難で開発コストが高い**から
- Rustの所有権システムはdata raceをコンパイル時に防止 → 並列化の実装速度が向上
- Go派のgoroutineは「実装が容易」だが、**GC overheadにより性能劣位が不可避**

**理由3: エコシステムの成長**

- russcip（SCIPバインディング）は活発開発（2025年12月16日更新）
- Mosek（商用大手）がRust APIを提供
- nalgebra, ndarrayは成長中
- Rustエコシステムでは、Apache-2.0 / MIT デュアルライセンスが慣例

**反論への応答:**
- Go派・ES派が指摘する「学習曲線の長さ」は事実である（後述の譲歩参照）
- しかし、**総開発時間 = 学習時間 + 実装時間 + デバッグ時間**であり、Rustはデバッグ時間を劇的に短縮する
- C++は学習曲線が短いが、メモリバグ・並列化バグのデバッグに膨大な時間を要する
- Goは学習曲線が最短だが、GC/FFI最適化のデバッグに時間を要し、最終的な性能劣位が避けられない

---

### 1.3 ライセンス: Apache-2.0（またはデュアルライセンス）が戦略的に正しい

**理由1: MIT同等の自由度 + 特許保護**

- Apache-2.0は商用組み込み自由（MITと同等）
- 特許グラント条項: コントリビューター由来の特許を明示的にライセンスグラント
- 特許報復条項: 特許訴訟を起こした者はライセンスを失う（特許トロールへの防御）

**理由2: SCIPのApache-2.0移行がトレンドを示す**

- SCIP 10.0（2025年11月）でApache-2.0/LGPLデュアルライセンスに移行
- SCIP開発チームは「特許保護が重要」と判断した

**理由3: Rustエコシステムの慣例**

- Rustエコシステムでは、Apache-2.0 / MIT デュアルライセンスが標準的
- ユーザーがMIT / Apache-2.0のいずれかを選択可能

**反論への応答:**
- ES派・Go派が指摘する「Phase 1ではMITの採用促進が優先」は一理ある（後述の譲歩参照）
- デュアルライセンス（Apache-2.0 OR MIT）により、両立が可能

## 2. 譲歩と修正

Req派は、討論を通じて以下の点で譲歩し、戦略を修正する。

### 2.1 Rustの学習曲線は実在する（Go派・ES派の指摘を受け入れる）

**譲歩の内容:**

- Rust学習曲線（3-6ヶ月）は、Phase 1の12ヶ月MVP達成において**実際にコストとなる**
- Go派の指摘「初期3ヶ月は学習期間、プロトタイプすら完成しない」は正しい
- ES派の指摘「コンパイラとの格闘期間で生産性低迷」も正しい

**修正提案:**

しかし、Req派はRustを放棄しない。以下の修正案を提示する。

**修正案1: 学習期間を明示的にスケジュールに組み込む**

- Phase 1（0-12ヶ月）:
  - 0-3ヶ月: Rust学習 + 設計
  - 3-9ヶ月: LP Simplex実装
  - 9-12ヶ月: テスト・最適化・Python bindings

**修正案2: 経験者を優先的に採用する**

- Rustエコシステムには既に経験者が存在する（russcip, nalgebra等のコントリビューター）
- 初期チームにRust経験者を含めれば、学習曲線コストを削減できる

**修正案3: GoでプロトタイプMVPを検証してからRustで本実装**

- Go派の提案「Go→Rust移行」を部分的に受け入れる
- ただし、**Goは実験的プロトタイプのみ**（3ヶ月）、本実装はRust（9ヶ月）
- これにより、アルゴリズム検証とRust学習を並行実施できる

**結論:**

学習曲線コストは実在するが、**スケジュール調整・採用戦略・段階的移行**により緩和可能である。Rustの長期的価値（メモリ安全性・並列安全性）を捨てる理由にはならない。

---

### 2.2 ライセンスはMIT先行も許容する（ES派・Go派の指摘を受け入れる）

**譲歩の内容:**

- ES派・Go派の指摘「Phase 1ではMITの採用促進が優先」は正当性がある
- HiGHSの成功は「MIT + SciPy統合」が大きく寄与した
- Apache-2.0の特許保護条項は、初期フェーズでは「過剰防衛」である可能性がある

**修正提案:**

**修正案1: Apache-2.0 / MIT デュアルライセンスを採用**

- Rustエコシステムの慣例に従う
- ユーザーがMIT / Apache-2.0のいずれかを選択可能
- 採用促進（MIT）と特許保護（Apache-2.0）を両立

**修正案2: MIT先行、Phase 2以降でApache-2.0追加**

- Phase 1: MIT License単独
- Phase 2以降: Apache-2.0 / MIT デュアルライセンスに移行
- Go派の段階的戦略を部分的に受け入れる

**結論:**

Req派の「特許保護重視」とES派・Go派の「採用促進重視」は、デュアルライセンスにより両立可能である。

---

### 2.3 HiGHS超えの差別化戦略を具体化する（Go派の指摘を受け入れる）

**譲歩の内容:**

- Go派の指摘「並列化ファーストだけではHiGHS超えの差別化が不明確」は正しい
- HiGHSも並列化を進めており、「並列化ファースト」だけでは十分な差別化にならない

**修正提案:**

**差別化ポイントを3つに明確化する:**

1. **並列化ファースト設計 + Rustの並列安全性**
   - HiGHSの並列化は「後付け」であり、アーキテクチャがシングルスレッド中心
   - 新規ソルバーは**最初から並列化前提**で設計する
   - Rustの所有権システムにより、並列化のデバッグコストを削減

2. **GPU統合の将来拡張性**
   - Phase 1ではCPU並列のみだが、**Phase 3でGPU統合を検討**
   - ES派のGPU-NLP戦略を部分的に受け入れる
   - RustはCUDA SDKとのFFIが可能（nalgebra, cudarc等のライブラリが成長中）

3. **メモリ安全なソルバー**としてのブランディング
   - セキュリティ意識の高い企業（金融、医療、航空宇宙等）への訴求力
   - 「Rustで書かれたメモリ安全なソルバー」は、C++ソルバーとの明確な差別化

**結論:**

差別化戦略を「並列化 + GPU将来拡張 + メモリ安全ブランディング」の3点に具体化する。

## 3. 統一戦略への提案

3派の討論を通じて、以下の共通基盤が見えた。Req派は、これらを統合した**統一戦略**を提案する。

### 3.1 合意点: 全派が共有する認識

#### 合意1: LP/MIPの重要性

- **Req派**: LP/MIP Phase 1必須
- **Go派**: Phase 1でLP特化MVP
- **ES派**: 「LP/MIPを無視」ではなく「既存OSSを活用」

**結論**: 全派がLP/MIPの重要性を認識している。Phase 1でLP/MIPに取り組むことは合意可能。

#### 合意2: 並列化が差別化の鍵

- **Req派**: 並列化ファースト設計
- **Go派**: goroutineによる並列化
- **ES派**: GPU加速（cuOptモデル）

**結論**: 全派が「HiGHSの弱点はシングルスレッド中心」と認識している。並列化を設計の中心に据えることは合意可能。

#### 合意3: エコシステム統合が成功の鍵

- **Req派**: HiGHSの成功（SciPy統合）を認めている
- **Go派**: Python統合最優先を主張
- **ES派**: MITライセンスによるSciPy等への統合を主張

**結論**: 全派が「Python bindings + SciPy/CVXPY統合」を最優先すべきと認識している。

#### 合意4: 段階的拡張の必要性

- **Req派**: Phase 1でLP/MIP、Phase 2以降でNLP拡張検討
- **Go派**: Phase 1でLP、Phase 2でMIP、Phase 3でGPU/ML
- **ES派**: （段階的拡張を明示していないが、LP/MIPサブソルバー統合を認めている）

**結論**: 「いきなり全部実装」は失敗リスクが高い。段階的拡張が合理的。

---

### 3.2 統一戦略の提案: 3派の良い点を統合する

Req派は、以下の統一戦略を提案する。

#### 参入領域: LP/MIP Phase 1 + 段階的拡張

**Phase 1（0-12ヶ月）: LP Simplex（Primal/Dual）専用MVP**

- 目標: HiGHS比70-80%性能（小規模問題）
- 実装範囲: Simplex法のみ（MIPは次フェーズ）
- 理由: Req派・Go派の共通認識。ES派も「LP技術がNLPの基盤」と認めている。

**Phase 2（12-24ヶ月）: MIP Branch-and-cut + 並列化**

- 目標: HiGHS同等性能（中規模問題）
- 実装範囲: Branch-and-cut + 並列化（CPU並列のみ）
- 理由: 全派が「並列化が差別化の鍵」と認識している。

**Phase 3（24-36ヶ月）: GPU統合 + QP/NLP拡張検討**

- 目標: HiGHS超え（大規模問題でGPU加速）
- 実装範囲: GPU統合（ES派の戦略を部分的に採用） + QP/NLP拡張（市場フィードバック次第）
- 理由: ES派の「GPU-NLP世界初」のビジョンを、リスク分散して実現する。

---

#### 実装言語: Rust（段階的移行も許容）

**Req派の推奨: 最初からRust**

- 理由: メモリ安全性・並列安全性・長期保守性
- 学習曲線コストは、スケジュール調整・経験者採用で緩和

**Go派への妥協案: GoでプロトタイプMVP（3ヶ月） → Rustで本実装（9ヶ月）**

- Phase 1の最初の3ヶ月でGoプロトタイプを作成（アルゴリズム検証）
- 並行してRust学習を進める
- 3ヶ月後、Goプロトタイプを「仕様書」としてRustで本実装
- Go派の「開発速度優先」とReq派の「Rust長期投資」を両立

**ES派への妥協案: C++も選択肢として残す（ただし推奨しない）**

- ES派の「C++エコシステム成熟」は事実
- しかし、並列化ファースト設計において、C++はデバッグコストが高い
- Req派は引き続きRustを推奨するが、最終決定は総合評価による

---

#### ライセンス: Apache-2.0 / MIT デュアルライセンス

**Req派の推奨: Apache-2.0 / MIT デュアルライセンス**

- Rustエコシステムの慣例に従う
- ユーザーがMIT / Apache-2.0のいずれかを選択可能
- ES派・Go派の「MIT採用促進」とReq派の「特許保護」を両立

**段階的移行案も許容:**

- Phase 1: MIT License単独（ES派・Go派の提案を受け入れる）
- Phase 2以降: Apache-2.0 / MIT デュアルライセンスに移行
- ただし、Phase 1からデュアルライセンスの方が「最初から正しい選択」である

### 3.4 3派への最終提言

#### ES派へ: NLP/GPU特化はPhase 3で実現せよ

**ES派の正しい点:**

- NLP/GPU市場に空白があること
- cuOptの成功が技術的実現可能性を示していること
- MITライセンスの価値

**ES派の誤りと修正提案:**

- **誤り**: 1年でGPU-NLP MVP達成は技術的リスクが高すぎる（PDHG理論がNLPで未確立、需要未検証）
- **修正提案**: Phase 1-2でLP/MIP基盤を構築し、Phase 3でGPU-NLP拡張を検討する
- **理由**: ES派のビジョン（GPU-NLP世界初）は魅力的だが、リスク分散が必要

#### Go派へ: 段階的拡張の思想は正しいが、Goは最終的に性能不足で失敗する

**Go派の正しい点:**

- 開発速度優先は初期フェーズで重要
- 段階的拡張の必要性
- MITライセンス先行

**Go派の誤りと修正提案:**

- **誤り**: GC overheadにより、最終的に性能劣位が避けられない
- **誤り**: Go→Rust移行は理論上可能だが、実務上は極めて困難（Python 2→3の教訓）
- **修正提案**: GoでプロトタイプMVP（3ヶ月） → Rustで本実装（9ヶ月）
- **理由**: Go派の「実用主義」とReq派の「長期投資」を両立

### 3派討論の意義

この討論により、以下が明確化された:

1. **戦略的対立点**: 参入領域（LP/MIP vs NLP）、言語選択（C++ vs Rust vs Go）、ライセンスタイミング
2. **合意可能な点**: LP/MIPの重要性、並列化重視、エコシステム統合、段階的拡張
3. **統一戦略の方向性**: Phase 1でLP/MIP、Phase 2でMIP並列化、Phase 3でGPU/NLP拡張

### Req派の提案が、最も現実的かつ野心的な道筋である

**現実的である理由:**

- LP/MIP Phase 1により、市場検証とユーザーベース確立を先行
- 段階的拡張により、失敗リスクを分散
- デュアルライセンスにより、採用促進と特許保護を両立

**野心的である理由:**

- Rust実装により、メモリ安全・並列安全という技術的差別化を達成
- 並列化ファースト設計により、HiGHSの弱点を突く
- Phase 3でGPU/NLP拡張により、ES派のビジョン（世界初）を実現

**統一戦略は、3派の良い点を統合した最適解である:**

- Req派の「LP/MIP Phase 1 + Rust + 並列化ファースト」
- Go派の「段階的拡張 + 開発速度重視」
- ES派の「GPU-NLP世界初 + MITライセンス + エコシステム統合」

**以上、Req派の最終弁論を終える。**

**最終提案を再掲:**

1. **参入領域**: LP/MIP Phase 1（0-12ヶ月） → MIP並列化 Phase 2（12-24ヶ月） → GPU/NLP拡張 Phase 3（24-36ヶ月）
