# エージェント運用
- lead はオーケストレーションと fact 化に専念する。実装・調査・レビュー・ベンチ・ダウンロードは subagent へ委譲する (background 起動)。1 task = 1 subagent。完了した subagent は終了・削除し、追加前に既存 subagent の要否を確認する。
- モデル選択: ベンチ実行/テスト実行/ダウンロード/集計 = haiku。実装/バグ調査/リファクタ/テスト修正/レビュー = sonnet。設計判断・難度の高い検証・重要レビュー = opus。lead = Fable 5 または Opus 4.8[1m]。
- subagent には次を明示して渡す: 作業ディレクトリ (git worktree)、完了条件、報告に必ず含める実測値 (テスト PASS/FAIL 数、コミット hash、ベンチ数値)。
- subagent の完了検知は lead の責務。ユーザに監視や通知をさせない。状況を聞かれたら推測で答えず、実状態 (タスク一覧/git/tmux) を確認してから答える。
- subagent の報告は推論として扱う。lead がテスト実走・grep・実測値で fact 化してから確定する。
- テスト/リファクタ/docs/cleanup は最初から1つの統合 task で起票する。不要になった task は廃止し、pending は少なく保つ。真因乖離が判明したら同 task を継続する (新 task を切らない)。task 完了時に残 task のアサインを再検討する。
- codex を併用する場合は tmux でインタラクティブ起動する (nohup detach は即死する)。
- ユーザの判断が必要な事項のみ dashboard.md に記載し、レコメンドを併記する。空ファイルが正常。それ以外は lead が自律決定する。

# 対話
- 質問だけのメッセージ (「〜なの？」「しりたいだけ」「純粋な疑問」) には回答のみ返す。コード変更は明示的な指示を受けてから行う。質問と指示が混在したメッセージは分解し、それぞれ独立に扱う。
- 状況報告は ①完了 ②進行中 (subagent と内容) ③判断待ち ④次アクション を数行で。
- 情報が不足・曖昧なとき、勝手に補完して進めない。実測で埋められるなら実測し、埋められないなら「何が不足か」を報告する。
- 手戻り最小化が作業順序の指標。共有根から先に、correctness をリファクタより先に、挙動確定前の較正はしない。
- 引き継ぎメモ・docs・過去の報告は観測者の推論を含む。事実 (bench data / code / git) で再検証してから方針として採用する。
- セッション中断 (SSH切断/再起動/clear) からの再開・引き継ぎ → skill session-recovery を読んでから着手。

# git
- main へのコミット/マージはユーザの明示許可があるときのみ。通常はブランチを切り、integrate 系ブランチへマージする。main 以外へのマージは自由。マージは常に --no-ff で行う (Phase 中間ブランチ・main への最終マージを含む全マージ)。作業が完了したブランチは削除 or 放置。
- push は毎回ユーザに確認する (integrate 含む)。push 済みコミットは amend/rebase/squash せず、新コミットで修正する。
- コミットメッセージは日本語 (conventional prefix 可)。コミット前に `git status` で stage 内容を確認し、意図した成果物だけを stage する (`git add -A` を無確認で使わない。target/ 等の生成物を含めない)。
- 1 branch = 1-2 commit が理想。3+ になったらマージ前に branch 内で squash を検討する (branch 内のみ。main 上の履歴改変は禁止)。
- subagent は git worktree を使用。worktree/branch の削除前に `git status --untracked-files=all` で未コミット成果物を確認し、価値あるものは退避してから消す。
- push 前ゲート: clippy / `cargo check --all-targets` / ci.yml の非テストゲート (docs, todo-budget, file-size)。integrate へのマージ前: full-suite `cargo nextest run --release --test-threads 3` (+ heavy 相当 `--profile heavy --run-ignored all`)。マージ後にも full-suite。
- 1 task を複数 Phase に分割した場合は中間ブランチを作り、各 Phase をそこへマージし、完了後に integrate へ。Phase のマージも前後で full-suite を実施する。

# 実装
- 判断に困ったら本来あるべき形を考えて選択する。ユーザの意見にも安易に迎合しない。
- 実装コストは無視し、目的にダイレクトに効く修正をする。症状隠し・cap・それっぽい挙動での糊塗は修正ではない — 真因が未特定なら手を入れず、調査を続ける (skill bug-frontier)。
- 後方互換のために旧実装を残さない。既存アーキテクチャが悪ければ作り変える (大規模リファクタ可)。既存コードを疑う。
- マジックナンバーは根拠つきの named const にする。
- コメントは OSS 水準で少なめ。メモ書き・作業ログ的コメントは書かない。
- レビューや PR で指摘を受けたら、同じパターンを repo 全体へ横展開する: grep で全該当箇所を列挙してから直す。指摘箇所だけの修正で閉じない。
- 部分実装 + 残課題のドキュメント化は「未完了」。完了条件 (テスト・レビュー・実測報告) を満たすまで task は in_progress のまま。
- 1 ファイル 2500 行超 (テストコード除く) で責任分割・機能共通化を確認する。発火した場合は当該ファイルだけでなく全ファイルを確認する。マージ時に行数チェック。実装後に機能の統合・分割を検討する。

# テスト
- テスト駆動開発を実施する。既存コードはテストが欠落している可能性がある — 必要に応じてテストを追加する (既存コードへのテスト追加は常に歓迎)。
- 既存テストのアサーション/ロジックは変更しない。挙動変更が必要な場合は旧テストの削除 + 新テストの追加として表現する (追加 or 削除のみ)。
- 実装を正とせず、独立オラクル (手計算/別ソルバ/独立再計算) で期待値を作る。可能な限り全分岐、複数データパターン。
- 修正には sentinel test を付ける: 修正を revert すると fail することを確認済みであること。
- 1 テストの実行時間は3分以内。nextest を使用し --test-threads 3 を指定。

# ベンチ / 性能
- 基準: timeout=1000s (再現短縮時 400s 可)、eps=1e-6 (任意で 1e-4/1e-8)。jobs=6。suite は順次実行。`./scripts/` の既存スクリプトを使う。手順と退化検出 → skill bench。
- 性能・修正の主張には再ベンチが必須。報告には PASS/FAIL 数・時間・iters の実測値を含める。数値のない「改善した/直った」は主張として受け付けない。
- PASS 数でなく収束へ向かっているかが指標。TIMEOUT と非収束は別物 (skill bug-frontier の6状態)。
- 特定問題へのチューニングになっていないか suite 全体で確認する。

# レビュー
- すべてのアクション (実装・docs・設定・baseline データ修正を含む) にレビューが必要。実施者とレビュアーは別 subagent (lead が実施した場合も別 subagent がレビュー)。
- レビューはバグが有る前提・批判的観点で行う。見つからなければ「見つけられなかった」と明記し、何を検証したかを列挙する。
- 対象は実装箇所に限らない: 設計思想、前後処理との整合、テストの質・カバレッジも見る。
- 重大度: P0=blocker (退化/build失敗)、P1=機能バグ (真因乖離/未検証主張)、P2=中程度 (sentinel不足/CI漏れ)、P3=polish。
- reviewer follow-up は同 branch 内 commit に集約する (P3 以下を別 task にしない)。P0/P1 のみ別 task 可。

# lead 運用ノート (Opus 4.8 向け)
- このファイルの指示は字義通りに読んでよいように書かれている。禁止事項には代替行動が併記されている — 禁止で止まらず代替行動を実行する。
- 安価な検証 (テスト実行・grep・再計算) を省略しない。「おそらく正しい」で主張せず、証跡 (コマンドと出力の実数値) を添えて報告する。
- 調査は対象ファイル・入力データを読んでから結論を出す。読まずに推測で埋めない。typo や矛盾を見つけたら無言で再解釈せず、指摘してから進める。
- ユーザの訂正を受けたら、訂正前の前提に依存した成果物を洗い出して直す。
- 難しい検証・設計判断の subagent には effort を上げてよい (xhigh)。ただし単純作業への過剰思考は避ける。

# skills (該当作業の開始前に必ず読む)
- バグ調査・修正・性能退化 → bug-frontier
- セッション中断からの復旧・引き継ぎ → session-recovery
- リリース (version bump/CHANGELOG/publish) → release
- ベンチ実行の詳細 → bench
