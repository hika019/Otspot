# git
- mainへコミットするな
- mainへマージはffを使うな.
- エージェントはgit worktreeを使用

# 方針/実装
- ユーザーの意見に安易に迎合するな
- 実装コストは無視し、やりたいことに対して、ダイレクトに効くものを実装せよ。上っ面の修正で隠したり、それっぽい動きをするようにするな
- 既存のコードを疑え
- テストは可能な限り全分岐を行え。実装を正とするな。複数パターンのデータを用意せよ。
- failを「範疇外」「別の真因」で分離するな。検証空白 (no_ref / 合成データのみ / SKIP) はbug不在を保証しない。空白を埋めるテスト追加 or 真因対処で応答せよ。
- マジックナンバーは排除しろ。バグの温床。
- テスト1つの実行時間を3分以内にせよ。cargo testが長時間であってはならない。
- テスト駆動開発を行え。既存コードに対してテストを追加して良い。
- マイクロアーキテクチャ
- 大規模リファクタリング可
- コメントが過多. OSSとしてふさわしい程度にしろ. むしろ少なめでよい
- 判断に困ったら本来あるべき形を考え、選択せよ
- コードにメモ書きコメント禁止

# レビュー
- すべてのアクションにはレビューが必要. 
- 実施者とレビュアーは別エージェント. 
- 批判的観点とバグが有る前提でレビューを実施. バグを見つけることができなかったのであればそのまま報告.
- 実施箇所も含めた全体としてレビューを行え. 設計思想自体が良くないなど全体としてのレビューや前後処理の不一致など実装箇所以外もレビュー対象とする. テストもレビュー対象とする.

# バグ対応
- バグ対応は入力から順に疑え
- 既存のコードを信じるな
- 暫定対処ではなく、原因に対してアクションを行え
- マジックナンバーや多数の分岐は悪
- 事実と推論は明確に区別せよ。推論に対してアクションをして事実に昇格させよ
- 引き継ぎメモ・docsは観測者の推論を含む。事実 (bench data / code) で再検証してから方針を採用せよ。
- PASS数にこだわり目的を見失うな
- バグ修正によって、他のバグが表面化することはよくある。安易に決めつけるな
- 既存コードはテストが欠落している可能性がある. 必要に応じて追加

# 実行
- `./scripts/` を使用するなど効率よく進めよ
- 実行には時間がかかるため、原則cpu600%を上限とし、並列でベンチをするなど効率化を行え
- ベンチの基準はtimeoutが1000sec, eps=1e-6を基準とする。ただし、1e-6にチューニングされていないことを確認するため、任意でeps=1e-4, 1e-6, 1e-8の3パターンでも実行せよ。
- PASS数ではなく順調に収束へ向けて探索が行えているかを確認せよ。
- PCのリソースを考え、各問題集は順次実行せよ
- テストはnextestを使用
