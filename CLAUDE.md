# git
- mainへコミット禁止(ユーザ許可の場合は除く)
- マージはffを使うな
- main以外へのマージは自由
- エージェントはgit worktreeを使用
- 作業が完了したブランチは削除 or 放置
- 1 branch = 1-2 commit 理想. 3+ で merge 前 squash 検討 (logical unit 集約). main 上の commit 履歴改変禁止 (rebase/squash/amend は branch 内のみ).
- mainへマージする前にtestを実行.
- 1タスクを複数Phaseに分割した場合, 中間ブランチを作成し、そこへ各Phaseをマージ. Phase終了後mainへマージ

# 方針/実装
- *判断に困ったら本来あるべき形を考え、選択*
- ユーザーの意見に安易に迎合するな
- 実装コストは無視し、やりたいことに対して、ダイレクトに効くものを実装せよ。上っ面の修正で隠したり、それっぽい動きをするようにするな
- 既存のコードを疑え
- テストは可能な限り全分岐を行え。実装を正とするな。複数パターンのデータを用意せよ。
- failを「範疇外」「別の真因」で分離するな。検証空白 (no_ref / 合成データのみ / SKIP) はbug不在を保証しない。空白を埋めるテスト追加 or 真因対処で応答。
- マジックナンバーは排除。バグの温床。
- テスト1つの実行時間は3分以内が目安。cargo testが長時間はNG。
- テスト駆動開発を実施。既存コードに対してテストを追加OK。
- testは原則追加 or 削除のみ. 変更は禁止. 本来関数の出力結果が変化することはない.
- マイクロアーキテクチャ
- 大規模リファクタリング可
- コメントが過多. OSSとしてふさわしい程度にしろ. むしろ少なめでよい
- コードにメモ書きコメント禁止
- 1ファイルが2500行以上(テストコードは除く)の場合、マイクロアーキテクチャとなっているか確認. マイクロアーキテクチャチェックが発火した場合は全ファイルを確認. 責任分割, 機能共通化, テストを追加を確認.
- マージ時に行数チェック
- 実装後に機能の統合・分割を調査, 検討, 実施

# レビュー
- すべてのアクションにはレビューが必要. 
- 実施者とレビュアーは別エージェント. 
- 批判的観点とバグが有る前提でレビューを実施. バグを見つけることができなかったのであればそのまま報告.
- reviewer follow-up は同 branch 内 commit に集約 (P3 以下は別 task 化禁止). P0/P1 重大バグのみ別 task 可.
- 重大度: P0=blocker (退化/build失敗), P1=機能バグ (真因乖離/未検証主張), P2=中程度改善 (sentinel不足/CI漏れ), P3=細部 polish (コメント/named const/docstring).
- アーキテクチャ, 設計Review
- 実装Review
- テストのカバレッジReview
- 実施箇所も含めた全体としてレビューを行え. 設計思想自体が良くないなど全体としてのレビューや前後処理の不一致など実装箇所以外もレビュー対象とする. テストもレビュー対象とする.
- 任意: 行数チェックと分割案/リファクタリング案の提案

# バグ対応
- バグ対応は入力から順に疑え. 発生場所と原因は一致しない事がある. 問題の読み込みから順に疑え
- 既存のコードを信じるな
- 暫定対処ではなく、原因に対してアクションを行え
- マジックナンバーや多数の分岐は悪
- 事実と推論は明確に区別せよ。推論に対してアクションをして事実に昇格させよ
- 引き継ぎメモ・docsは観測者の推論を含む。事実 (bench data / code) で再検証してから方針を採用せよ。
- PASS数にこだわり目的を見失うな. 収束しているかが指標
- バグ修正によって、他のバグが表面化することはよくある。安易に決めつけるな
- 既存コードはテストが欠落している可能性がある. 必要に応じて追加

# 事実積み上げ (検証フロンティア) — バグ対応の基本手法
- 全バグ対応の基本。対象を入力→出力の処理段に分解し、各段の正当性を入力から順に「事実」として確定する。前段が proven でなければ下流の正誤は判定不能 (発生場所≠真因)。proven 判定は推論禁止・実測のみ。
- 処理段は対象ごとに定義 (どんな対象も「入力 → 変換段 … → 出力」に分解)。例: LP = parse→presolve→Phase I→Phase II→postsolve。QP = presolve→IPM(IPPMM)反復→postsolve。MIP/非凸QP は各 B&B ノードが LP/QP を呼ぶ土台依存。
- フロンティア = proven 済の最下流の「次の段」。検証も修正も常にフロンティア段に対して行う。上流が未 proven のまま下流症状を追うな。
- 各段で全種のバグを洗い出す (片方だけ見るな、複数バグ前提): correctness (誤った出力 / 符号反転 / 誤った変換・縮小 / 不変条件違反 / 偽判定 false-Infeasible 等) と それ以外 (収束しない / hang / cycle / stall / 性能劣化)。
- proven の具体手段 (実測。推論で済ますな):
  - 入力段 = 内部表現を生入力と機械突合 (例 parse: 制約数/sense/RHS/bound/変数数)。
  - 中間変換段 = 入出力の等価性・不変条件を機械検査 + 全分岐 sentinel test (no-op で fail する設計) (例 presolve: ON/OFF/known 不変性スイープで符号・縮小誤りを検出)。
  - 計算段 = 出力を独立再計算 or 証明書で検証 (例 Phase I: 基底の実行可能性・Farkas・dual符号 / 最終解: 原制約充足・report obj == 再計算 obj・往復不変)。
- ループ (1 iteration):
  1. 観測を再現 (ソルバは bench: timeout=400 or 1000, eps=1e-6 固定) — Agent 実行。
  2. 異常を【段】×【correctness / それ以外】で分類 (実測値: timing/iters/残差 等から)。
  3. フロンティア段を厳密検証し proven か全バグかを確定 (Agent/worktree、env-gated トレース可・報告後 revert)。
  4. 見つけた全バグを真因修正 (本来あるべき形・マジックナンバー禁止・sentinel test。暫定 / cap / 症状隠し / 当該段の無効化 は禁止)。
  5. 別 Agent レビュー + `codex review --commit <SHA>` gate (実施者≠レビュアー)。
  6. lead-verify full-suite (`cargo nextest run --release --test-threads 3`) → integrate へ --no-ff マージ。
  7. フロンティア段の proven を再確認 → フロンティア前進 → 再観測 (次 iteration)。
- Agent 報告は推論。lead がテスト実走 / grep / 実測値で fact 化してから確定。
- 停止条件: フロンティアが出力到達 (全段 proven) / 要 user 判断 / 割込。

# 実行
- `./scripts/` を使用するなど効率よく進めよ
- 実行には時間がかかるため、原則job=6固定とし、並列でベンチをするなど効率化を行え
- ベンチの基準はtimeoutが1000sec, eps=1e-6を基準とする。任意でeps=1e-4, 1e-6, 1e-8の3パターン
- PASS数ではなく順調に収束へ向けて探索が行えているかを確認せよ。
- PCのリソースを考え、各問題集は順次実行せよ
- テストはnextestを使用. --test-threads 3を指定
