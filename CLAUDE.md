# claude code
- チームを使用. tmux使用
- ベンチ実行 or テスト実行 or downloadなどの軽量タスク=haiku, 実装 or リファクタ or テスト修正=sonnet, それ以外=opus. これを基準にAgentのmodelを設定
- タスクが一つでもエージェントに振れ
- エージェントのコンテキストを考え、タスクを分割・アサイン. Agentはbackground起動. tmux.
- エージェント追加時に、既存Agentの確認. 不要Agentの場合削除. kill可
- タスク追加時にタスクの統合を検討. テスト/リファクタ/docs/cleanup は最初から統合 task で作成 (個別 task 分割禁止). 前タスクで不要となったタスクは廃止.
- タスク完了時にタスクアサインを検討
- タスク完了後エージェントは終了&削除. 1タスク1エージェント
- 明示的な許可の場合はclaude.mdを編集
- ユーザの判断が必要な場合`dashboard.md`に記載. レコメンドを実施. 空ファイルが正常
- 真因乖離時 = 同 task 継続

# git
- mainへコミット禁止(ユーザ許可の場合は除く)
- マージはffを使うな
- main以外へのマージは自由
- エージェントはgit worktreeを使用
- 作業が完了したブランチは削除 or 放置
- 1 branch = 1-2 commit 理想. 3+ で merge 前 squash 検討 (logical unit 集約). main 上の commit 履歴改変禁止 (rebase/squash/amend は branch 内のみ).
- mainへマージする前にtestを実行.
- 1タスクを複数Phaseに分割した場合, 中間ブランチを作成し、そこへ各Phaseをマージ. Phase終了後mainへマージ

# 方針/実装
- *判断に困ったら本来あるべき形を考え、選択*
- ユーザーの意見に安易に迎合するな
- 実装コストは無視し、やりたいことに対して、ダイレクトに効くものを実装せよ。上っ面の修正で隠したり、それっぽい動きをするようにするな
- 既存のコードを疑え
- テストは可能な限り全分岐を行え。実装を正とするな。複数パターンのデータを用意せよ。
- failを「範疇外」「別の真因」で分離するな。検証空白 (no_ref / 合成データのみ / SKIP) はbug不在を保証しない。空白を埋めるテスト追加 or 真因対処で応答。
- マジックナンバーは排除。バグの温床。
- テスト1つの実行時間は3分以内が目安。cargo testが長時間はNG。
- テスト駆動開発を実施。既存コードに対してテストを追加OK。
- マイクロアーキテクチャ
- 大規模リファクタリング可
- コメントが過多. OSSとしてふさわしい程度にしろ. むしろ少なめでよい
- コードにメモ書きコメント禁止
- 1ファイルが2500行以上(テストコードは除く)の場合、マイクロアーキテクチャとなっているか確認. マイクロアーキテクチャチェックが発火した場合は全ファイルを確認. 責任分割, 機能共通化, テストを追加を確認.
- マージ時に行数チェック
- 実装後に機能の統合・分割を調査, 検討, 実施

# レビュー
- すべてのアクションにはレビューが必要. 
- 実施者とレビュアーは別エージェント. 
- 批判的観点とバグが有る前提でレビューを実施. バグを見つけることができなかったのであればそのまま報告.
- reviewer follow-up は同 branch 内 commit に集約 (P3 以下は別 task 化禁止). P0/P1 重大バグのみ別 task 可.
- 重大度: P0=blocker (退化/build失敗), P1=機能バグ (真因乖離/未検証主張), P2=中程度改善 (sentinel不足/CI漏れ), P3=細部 polish (コメント/named const/docstring).
- アーキテクチャ, 設計Review
- 実装Review
- テストのカバレッジReview
- 実施箇所も含めた全体としてレビューを行え. 設計思想自体が良くないなど全体としてのレビューや前後処理の不一致など実装箇所以外もレビュー対象とする. テストもレビュー対象とする.
- 任意: 行数チェックと分割案/リファクタリング案の提案

# バグ対応
- バグ対応は入力から順に疑え. 発生場所と原因は一致しない事がある. 問題の読み込みから順に疑え
- 既存のコードを信じるな
- 暫定対処ではなく、原因に対してアクションを行え
- マジックナンバーや多数の分岐は悪
- 事実と推論は明確に区別せよ。推論に対してアクションをして事実に昇格させよ
- 引き継ぎメモ・docsは観測者の推論を含む。事実 (bench data / code) で再検証してから方針を採用せよ。
- PASS数にこだわり目的を見失うな
- バグ修正によって、他のバグが表面化することはよくある。安易に決めつけるな
- 既存コードはテストが欠落している可能性がある. 必要に応じて追加

# 実行
- `./scripts/` を使用するなど効率よく進めよ
- 実行には時間がかかるため、原則job=6固定とし、並列でベンチをするなど効率化を行え
- ベンチの基準はtimeoutが1000sec, eps=1e-6を基準とする。任意でeps=1e-4, 1e-6, 1e-8の3パターン
- PASS数ではなく順調に収束へ向けて探索が行えているかを確認せよ。
- PCのリソースを考え、各問題集は順次実行せよ
- テストはnextestを使用. --test-threads 3を指定
